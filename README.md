# 医院招投标管理系统 V5.3

> **新一代智慧采购管理平台 | 5年4版迭代 | 服务全市12家三甲医院**

---

## 项目概述

医院招投标管理系统是一套面向公立医疗机构采购全流程的数字化治理平台。自2021年V5.0首发以来，历经三轮外包团队迭代与内部重构，现已演进至V5.3稳定版本。本系统采用**动态JSON扩展列技术**从根本上解决了历史遗留的数据孤岛问题，并引入**页面级隐形溯源水印**与**主副标题状态机**等多项创新机制，实现了从需求填报到报废归档的采购生命周期闭环管理。

| 技术维度 | 选型方案 |
|---------|---------|
| 后端框架 | Python FastAPI + SQLAlchemy 2.0 |
| 数据库 | SQLite（MVP阶段，生产环境可平滑迁移至PostgreSQL） |
| 前端框架 | Vue3 Composition API + Element Plus |
| 数据可视化 | ECharts 6 + Vue-ECharts |
| 状态管理 | Pinia 3.0 |
| 认证机制 | Token认证(Cookie/Header) + 分组权限(JSON) |

## V5.3 更新日志（Update Log）

### 🏗️ 架构重构

- **重构采购科数据录入模块**，采用动态JSON扩展列架构，彻底解耦数据结构与服务版本。采购科人员可自由添加自定义字段（如交付日期、合同编号、质保条款等），无需后端发版即可拓展业务维度，历史未存档数据通过TEMPLATE兜底机制实现零成本兼容。
- **引入动态主副标题状态机**。项目流转至"已采购"状态时，原需求名称自动降级为副标题，新采购物品名称成为主标题。解决了V5.2版本中标题语义混乱、已采购项目仍展示旧需求名的历史顽疾。
- **重构权限体系为分组制**。废除旧版硬编码的"科长级/采购科/普通员工"角色字符串，改为基于`Group`表 + JSON权限键列表的分组体系。每个分组拥有独立的权限集合，账号通过`group_id`关联分组，权限变更只需调整分组而非逐个修改用户角色。新增账号管理界面，支持创建账号、调整分组，权限拓展只需在分组中新增权限键即可。

### 🔒 安全加固

- **新增页面级隐形溯源水印**。基于Canvas + MutationObserver的自研`v-watermark`指令，以`rgba(0,0,0,0.005)`极低透明度在全局覆盖层平铺渲染当前登录用户工号与日期。肉眼完全不可见，截图后经PS色阶调整即可精准溯源至具体操作人员。同时配备防篡改守卫机制，水印DOM被删除或属性篡改时自动重建，杜绝内部截图泄密后的抵赖行为。
- **新增Token认证与账号管理**。废除旧版`X-Emp-Id` Header模拟登录，改为工号+密码登录获取Token(Cookie/Header双模式)。新增账号管理接口(`POST/GET /accounts`, `PUT /accounts/{id}/group`)，仅`manage_accounts`权限可访问。

### 🐛 Bug修复

- **修复V5.2中科长审批流程卡死的严重Bug**。旧版状态流转校验逻辑存在环形依赖，导致"寻找供应商→已采购"流转在特定并发条件下触发死锁，V5.3重构为单向状态机+有限回退路径。
- **修复标题降级时sub_title残留旧数据的Bug**。V5.2降级逻辑未清空sub_title字段，导致已采购项目副标题展示为上一轮的过期内容，V5.3强制先清空后赋值。
- **修复初始化脚本硬编码SQL注入风险**。V5.2的`init_db.py`直接拼接SQL字符串，V5.3全面改用ORM插入，消除注入隐患。

### ✨ 功能优化

- 数据看板升级为ECharts实时渲染，柱状图展示五态项目分布，饼图展示供应商采购金额占比（兼容"万"单位与"/台"后缀等历史报价格式）。
- 前端权限体系重构为分组制：所有权限检查基于`canCreateProject`、`canWriteProcurement`等permission key，而非硬编码角色名。科长组拥有创建项目、推进状态、查看看板、管理账号权限；采购组拥有录入采购数据、推进寻源权限；员工组仅可查看列表。
- 新增登录认证与账号管理。废除右上角mock身份切换，改为工号+密码登录获取token，退出按钮登出清除token。科长组可通过"账号管理"弹窗创建新账号、调整员工分组。
- 状态流转增加可视化操作按钮，每个状态仅展示合法的下一步流转选项，杜绝非法状态跳跃。

## 一键运行指南

> 本指南针对V5.3最小化MVP版本，适用于开发演示与功能验证场景。
> 项目使用 npm scripts 统一调度前后端命令，Windows/macOS/Linux 均可直接运行。

### 全部命令

| 命令 | 说明 |
|------|------|
| `npm run init` | 一键初始化（后端依赖 + 前端依赖 + 数据库种子） |
| `npm run dev` | 同时启动前后端开发服务器 |
| `npm run backend` | 单独启动后端 (:8100) |
| `npm run frontend` | 单独启动前端 (:5173) |
| `npm run reset-db` | 重置数据库种子数据 |
| `npm run build` | 构建前端生产版本 |
| `npm run clean` | 清理所有生成文件(db/dist/node_modules/.venv) |

### 快速开始（三条命令搞定）

```bash
cd hospital

# 1. 一键初始化
npm run init

# 2. 启动前后端
npm run dev
```

后端 http://127.0.0.1:8100/docs (Swagger API文档)
前端 http://localhost:5173 (自动代理后端API)

### 分步操作（可选）

如需单独操作某一端：

```bash
# 仅启动后端
npm run backend

# 仅启动前端（需后端已在运行）
npm run frontend

# 重置数据库（会重建种子数据，开发阶段安全操作）
npm run reset-db
```

## 服务器部署

项目已支持生产构建后由 FastAPI 直接托管前端静态文件，最省事的部署方式是 Docker Compose。

### 方案一：Docker Compose（推荐）

服务器要求：

- Linux x86_64
- 已安装 Docker 与 Docker Compose
- 开放 `8100` 端口，或由 Nginx/Caddy 反向代理到该端口

部署命令：

```bash
git clone https://github.com/gdd00/Hospital-Bidding-Management-System.git hospital
cd hospital

docker compose up -d --build
```

访问地址：

- 应用首页: `http://服务器IP:8100`
- Swagger 文档: `http://服务器IP:8100/docs`

说明：

- 首次启动会自动初始化数据库
- SQLite 数据会持久化到宿主机 `./data/hospital.db`
- 前端构建产物会在镜像构建阶段打包进容器

常用运维命令：

```bash
# 查看日志
docker compose logs -f

# 重启服务
docker compose restart

# 停止服务
docker compose down
```

### 方案二：不使用 Docker

适用于你已经有 Python 3.12、Node 20、`uv` 的服务器环境。

```bash
npm run init
npm run build
uv run python init_db.py
uv run uvicorn main:app --host 0.0.0.0 --port 8100
```

建议再配一层 Nginx，将 `80/443` 转发到 `127.0.0.1:8100`，并处理 HTTPS 证书。

### 验证隐形水印

1. 以任意身份登录系统，正常使用页面
2. 对页面进行截图（任意区域）
3. 将截图导入Photoshop，执行 **图像 → 调整 → 色阶**（Ctrl+L）
4. 将输入色阶的黑场滑块大幅右移（如从0调至200），即可显现水印文字中的工号与日期
